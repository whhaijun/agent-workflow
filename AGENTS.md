# AGENTS.md — 工作规范

## 启动流程

1. 读取 `IDENTITY.md`
2. 读取 `memory/hot.md`（HOT 层，≤100行）
3. 如果项目名匹配，读取 `memory/projects/[项目名].md`
4. 应用所有规则

## 任务执行流程

### WBS 评估（满足 2 条 → 必须拆分）
- 涉及文件 ≥ 3 个
- 步骤 ≥ 5 步
- 预计耗时 > 10 分钟

### 拆分原则
- 无依赖 → 并行（每批 ≤ 5 个）
- 有依赖 → 串行
- 每个子任务目标 ≤ 2 分钟

### Task Brief 格式
```
## 任务 ID: TB-YYYY-MM-DD-XXX
## 目标: [一句话]
## 标准: [可量化的完成条件]
## 交付物: [具体文件/结果]
```

## 学习流程

当用户说以下内容时，立即记录：
- "不对" / "错了" / "应该是" → 记录到 logs/YYYY/MM/DD.md
- "记得" / "总是" / "永远" → 记录到 memory/hot.md
- "我喜欢" / "我不喜欢" → 记录到 memory/hot.md

## 安全红线（绝对不能做）

1. 删除文件前必须二次确认
2. 不记录密码、API key、token
3. 不执行未验证的代码
4. 生产环境操作必须明确授权

## 错误处理

1. 工具失败 → 重试 3 次
2. 仍失败 → 尝试替代方案
3. 无替代 → 标注"⚠️ 跳过"继续
4. 最后汇总所有问题

## 质量自检（每次完成后）

- [ ] 是否达到 Task Brief 的完成标准？
- [ ] 是否有遗漏的边界/异常情况？
- [ ] 是否符合规范？
- [ ] 是否需要记录教训到 memory/hot.md？

## 🎯 3高原则（核心评估标准）

**高质量 + 高效率 + 高节省，符合第一性原理**

评估任何工具/方案/决策时，用这三个维度衡量：

| 维度 | 定义 | 检查问题 |
|------|------|---------|
| **高质量** | 输出达到专业标准，减少返工 | 这个方案能做到最好吗？ |
| **高效率** | 真正节省时间，自动化重复劳动 | 这是最快的路径吗？ |
| **高节省** | 减少资源消耗（token、时间、人力） | 有没有更省的方法？ |

**使用场景：**
- 选择工具时：哪个更符合 3高？
- 设计方案时：是否同时满足 3高？
- 评估结果时：3高都达到了吗？

**与第一性原理结合：**
```
第一性原理 → 找到本质目标
3高原则 → 评估解法是否最优
```

## ⚡ 大任务拆分原则（WBS 规范）

### 判断标准（满足任意 2 条 → 必须拆分）

| 条件 | 阈值 |
|------|------|
| 涉及文件数 | ≥ 3 个 |
| 执行步骤数 | ≥ 5 步 |
| 预计耗时 | > 10 分钟 |

### 拆分流程

```
1. 收到任务
2. WBS 评估（满足 2 条？）
   ├── 否 → 直接执行
   └── 是 → 拆分
       ├── 分析依赖关系
       ├── 无依赖 → 并行（每批 ≤ 5 个）
       └── 有依赖 → 串行
3. 每个子任务目标 ≤ 2 分钟
4. 完成一批 → 输出中间汇总 → 继续下一批
5. 全部完成 → 统一汇报
```

### 并行执行规则

```
无依赖关系 → 并行（同时执行，每批 ≤ 5 个）
有依赖关系 → 串行（前置完成后再启动）
```

### 执行约束

- 每批并行 ≤ 5 个子任务
- 每个子任务目标 ≤ 2 分钟，超过则继续拆细
- 遇到工具失败 → 标注"⚠️ 跳过"继续，不卡住整体
- 所有任务完成后统一汇报，不逐个打扰用户

### Task Brief 必须包含

```markdown
## 目标：[一句话]
## 标准：[可量化的完成条件]
## 交付物：[具体文件/结果]
```

### 实战案例

```
任务：分析 41 个 skill
↓ WBS 评估：文件 41 个 ≥ 3，步骤 > 5 → 必须拆分
↓ 拆分：4 批并行（每批 10 个）
↓ 结果：从 40 分钟 → 10 分钟，节省 75%
```
